0-to-1 产品 AI 产品 差异化：Git 式简历版本控制

一、需求背景

一个认真求职的技术人手里从来不是「一份简历」，而是十几份针对不同公司、不同方向裁剪过的版本。三个月投下来，最好的项目经历散落在各处，基础信息改一次要手动同步十几个文件，最后连「腾讯那版到底突出了什么」都记不清。现有工具要么帮你把一份简历排版好看（Reactive Resume、OpenResume），要么帮你把简历对单个 JD 打分匹配（Teal、Huntr），没有人帮你管理这整片「简历森林」的版本演化。

核心痛点：多职位投递时，个人简历与岗位需求不匹配，且多版本之间混乱、难追溯、难复用。

证据强度：[PM 假设 / 个人痛点] — 待用户验证补充真实求职场景数据。

不解决的后果：求职者反复手工裁剪简历、丢失优质素材、无法沉淀投递经验，每次投递都从零开始。

二、产品定位与差异化

把简历当代码仓库来管：Master 主干分化出方向分支，分支长出公司专属节点，改一次主干所有子分支自动继承。

Resume-Agent 的差异化在于管理「森林」而非生成「单棵树」。React Flow 可视化的 Git 树是产品的灵魂一击——目标用户（算法、安全、后端工程师）天天活在 Git 里，把简历当分支管理是他们的母语，这是模板工具和匹配工具都抄不走的身份认同。

赛道代表产品它们做什么Resume-Agent 的差异

三、目标用户

  • 技术门槛人才：算法工程师、网络安全研究员、核心后端研发、芯片工程师。熟悉 Git 工作流，对「分支/合并/Diff」概念天然亲切。
  • 高质量求职者：研究生、大厂跳槽人员、追求高成功率的精准投递人群。同时投递多个方向、多家公司，版本管理需求强烈。

使用前提：用户有至少一份历史简历，且愿意把求职素材沉淀进系统。隐私敏感——倾向本地优先存储。

四、整体架构：三栏布局

系统采用左—中—右三栏结构，三块功能独立、操作高效。中间栏最宽，承载核心的 Git 树画布；左栏窄，负责导航与资产输入；右栏中等，负责对外职位分析与反馈。下图为布局线框，标注了各区域组件与信息层级，供前端设计直接参照。

五、功能需求（按栏划分）

5.1 左侧栏 — 导航与个人资产知识库（RAG 底座）

左栏是整个产品的「燃料仓」，建立基于检索增强生成（RAG）的个人履历智能检索系统。

#功能描述
A1导航系统提供「总览面板 / 简历版本分支 / 个人知识库(RAG) / 职位截图分析 / 技能差距分析 / 设置 / 用户支持」入口，当前所在项高亮。点击切换中栏与右栏的对应视图。
A2资产冷启动用户拖入所有历史版本旧简历（PDF/Word）。系统反向解构出原子级经历块，智能去重、合并、清洗，自动构建初始知识库与初始树状图（生成 Master 主干 + 推断的方向分支）。
A3资产养料注入支持上传代码日志、项目周报、学术论文、CTF 比赛报告或博客链接，作为知识库的补充素材。
A4智能存储用向量数据库存储上述信息切片，支持语义检索。优先本地向量库或私有化部署，确保隐私安全。

前端要点：左栏顶部固定 Logo 与导航列表；中段两个拖拽上传区（旧简历 / 知识素材）需有明显的拖入悬停态；底部常驻知识库索引状态指示器（切片数 / 索引进度）。

5.2 中间栏 — 树状版本控制（Resume Git）与智能生成

中栏是「中央枢纽」，用可视化方式管理简历演化，并按岗位动态组装。React Flow 树状画布是产品灵魂，需可缩放、可拖动、节点可点击进入编辑。

#功能描述
B1树状版本画布类 Git 树的可缩放界面。一个 Master 主库分支分化出不同方向分支（安全岗、算法岗），再分到公司专属节点（Tencent-研究员）。节点形态区分主干/方向/公司节点。
B2上游继承在 Master 主库修改基础信息，所有子分支自动集成更新，同时保留各子版本的公司专属描述不变。
B3按需动态生成选定一个岗位投递时：AI Agent 先从知识库检索相关经历；AI 面试官 Agent 反思审核经历是否真实（避免假大空）；AI 撰写 Agent 动态拼装并润色，生成卡在 1 页内的公司专属 PDF。
B4模板管家简历数据与排版分离（JSON-to-Resume）。一键在「经典学术 / 大厂技术 / 极简风」等 ATS 友好模板间切换，PDF 预览实时同步。
B5版本对比清晰展示某公司版简历相较另一版做了哪些项目替换与关键词突显。

前端要点：画布占中栏主体高度；顶部面包屑显示当前节点路径（Master → 安全岗 → Tencent-研究员）；底部工具条放「为该岗位动态生成」「版本对比 Diff」主按钮与模板切换器；PDF 预览区可折叠或侧拉。

5.3 右侧栏 — 职位截图分析与 AI 求职导师

右栏是「对外接口」，分析外部招聘信息并对内部能力给出提升建议。

#功能描述
C1多模态截图提取用户上传招聘 App、朋友圈海报或网页的招聘截图。AI 自动识别并结构化提取 JD 中的技术栈、软硬技能要求与加分项。
C2技能 Gap 分析结合 C1 提取的岗位要求与个人知识库，生成不编造、不作弊的《知识盲区报告》。
C3导师端反馈当 JD 要求的某技术点在知识库未发现时，AI 导师明确指出差距，并反向输出面试前补充学习建议（含可参考的开源项目/资料方向）。

前端要点：右栏自上而下三段——截图上传区、JD 结构化卡片、盲区报告 + 导师建议。盲区报告用三色状态标记（已覆盖 / 部分缺口 / 未涉及），导师建议可展开折叠。

六、核心交互流程

1
资产冷启动

用户拖入多版旧简历 → 反向解构为原子经历块 → 去重合并清洗 → 生成初始知识库与初始 Git 树（Master + 推断方向分支）。

2
分支创建与继承

从 Master 或方向分支新建公司节点 → 继承上游基础信息 → 用户仅编辑公司专属描述，基础信息由上游统一维护。

3
按岗位动态生成

选定公司节点 → 上传/粘贴该岗位 JD 截图 → AI 检索知识库相关经历 → 反思审核真实性 → 撰写润色生成 1 页 PDF。

4
差距反馈与学习

右栏生成知识盲区报告 → AI 导师输出面试前补充学习建议 → 用户沉淀新知识回灌知识库。

七、技术方向参考（非约束性建议）

以下为产品愿景中提及的技术方向，仅作设计/实现参考，不作为需求约束。最终技术选型由工程阶段决定。

层面建议方向理由
Agent 框架LangGraph 或 LangChain契合树状控制的图状工作流，支持反思与纠错机制
向量库本地优先 Chroma 或 PGVector隐私优先，适配技术用户对数据本地化的诉求
前端可视化React Flow成熟的节点图库，适合 Git 树画布的缩放与交互

关于「AI 审核经历真实性」：LLM 无法真正验证经历是否客观发生过，只能检测 AI 套话、前后矛盾与夸大表述。需求文案中「审核经历是否真实」应理解为「检测套话与一致性」，避免对用户过度承诺。

八、给前端设计的提示

  • 视觉调性：面向技术用户的开发者工具感，深色为主、信息密度较高、等宽字体点缀（呼应代码/Git 语境）。
  • 灵魂一击：中栏 Git 树画布是 README 截图、演示动图的核心画面，需最具视觉表现力——节点形态、分支连线、继承箭头要有设计感。
  • 三栏比例：建议左 18% / 中 52% / 右 30%，中栏最大化承载画布；窄屏可折叠左右栏为抽屉。
  • 状态完备：每个区域都需考虑空状态（无简历/无截图/无知识库）、加载态（解构中/生成中/索引中）、错误态（解析失败/生成失败）。
  • 待对齐细节：模板具体样式、Git 树节点交互细节、Diff 视图呈现方式、多端适配策略，留待后续与设计逐项确认。
